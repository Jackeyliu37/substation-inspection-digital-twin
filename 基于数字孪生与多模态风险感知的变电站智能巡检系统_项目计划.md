# 基于数字孪生与多模态风险感知的变电站智能巡检系统——项目计划

> **执行说明：** 本文是可移交给开发者或 AI 编程代理的项目级规格。实施时应先建立 Git 仓库和 `AGENTS.md`，再按“环境基线→仿真→感知→风险融合→任务规划→可视化→验收”的顺序推进；每一阶段都必须形成可独立运行、可测试的成果。

**文档版本：** 1.0
**编制日期：** 2026-07-22
**目标平台：** Ubuntu 24.04 LTS 服务器（AMD Ryzen 5 5500、NVIDIA RTX 3060 Ti、约 16 GiB 内存）
**核心技术栈：** ROS 2 Jazzy、Gazebo Harmonic（OGRE2/EGL无头渲染）、TurtleBot3 Waffle Pi、Nav2、SLAM Toolbox、PyTorch、Ultralytics YOLO11n、Next.js、React、TypeScript、FastAPI、Three.js、ECharts、Foxglove Web

## 1. 项目名称与题目含义

项目名称为“基于数字孪生与多模态风险感知的变电站智能巡检系统”。它不是单纯让机器人在 Gazebo 中移动，也不是只在摄像头图像上运行目标检测，而是把以下四类能力组成闭环：

1. **数字孪生：** 在 Gazebo 中建立包含变压器、断路器、隔离开关、绝缘子、仪表、围栏和巡检通道的虚拟变电站，并在 ROS 2 中维护每台设备的身份、位置、状态和风险等级。
2. **多模态风险感知：** 同时利用 RGB 图像、激光雷达、温度、烟雾和气体传感数据，不依赖单一视觉结果判断风险。
3. **风险驱动巡检：** 机器人不只按照固定路线巡航，而是根据设备风险分数调整巡检顺序；高风险设备出现时，可中断普通任务并重新规划路径。
4. **结果闭环：** 系统自动保存检测证据、机器人轨迹、设备风险变化和告警记录，并生成巡检报告。

项目的核心创新点是把“语义数字孪生—多模态风险融合—巡检任务重排序—Nav2 路径重规划”连成一个可以在仿真中验证的完整系统。

## 2. 项目目标与范围

### 2.1 总体目标

在 Ubuntu 24.04 服务器上完成一套可重复部署的 ROS 2 变电站巡检仿真系统。TurtleBot3 Waffle Pi 在 Gazebo Harmonic 变电站场景中执行自主巡检，通过 YOLO11n 和模拟环境传感器识别人员防护、烟火和设备目标，计算设备风险，并触发巡检优先级调整、路径重规划、告警记录及自动报告。

### 2.2 本期必须完成的功能

- 建立至少 8 类典型变电站设备及巡检通道的 Gazebo 世界。
- 建立 TurtleBot3 Waffle Pi 的 RGB 摄像头、二维激光雷达及模拟温度、烟雾、气体传感器。
- 完成 SLAM 建图、地图保存、定位和 Nav2 多目标点巡检。
- 使用 Ultralytics YOLO11n 识别人员/PPE、烟火和变电站设备。
- 建立以设备 ID 为中心的语义数字孪生状态表。
- 将视觉、温度、烟雾、气体和设备上下文融合为 0～100 风险分数。
- 根据风险等级动态调整巡检点优先级并重新调用 Nav2 规划。
- 建立现代化 Web 综合控制平台，在浏览器中显示三维数字孪生、地图、机器人、检测框、风险设备和路径。
- 在 Web 平台中控制巡检启动/暂停/继续/停止、返航、目标点、路线重规划、仿真场景和紧急停止。
- Gazebo只以无头服务器模式运行；保留Foxglove Web、ROS CLI和rosbag2作为开发调试与故障诊断工具。
- 保存 rosbag2、检测结果、风险事件和任务结果，并生成 HTML 或 PDF 巡检报告。

### 2.3 本期不做的内容

- 不接入真实变电站控制系统，不向真实设备下发控制指令。
- 不追求热传导、烟气扩散等高精度物理仿真；温度和气体采用可配置的场景模型。
- 不开发面向真实生产环境的多租户、复杂角色权限、移动 App 或公网云平台；本期 Web 平台定位为单操作员、局域网内的完整演示与控制平台。
- 不以强化学习替代 Nav2；风险驱动体现在任务优先级与路径代价上，以保证可解释性和可复现性。

## 3. 推荐部署方式

### 3.1 结论

项目不要求所有操作界面都安装在服务器本地显示器上，但建议所有**核心运行组件集中部署在 Ubuntu 服务器**。客户端电脑只作为轻量访问终端。

| 位置 | 部署内容 |
|---|---|
| Ubuntu 服务器 | ROS 2、Gazebo、Nav2、SLAM、YOLO11n、GPU 推理、数据集、模型、SQLite、FastAPI ROS Web Gateway、Next.js 前端、Nginx、Foxglove Bridge、报告生成器 |
| Windows/其他客户端 | 日常使用只需现代浏览器访问服务器Web地址；SSH用于Codex维护，Foxglove Web用于ROS诊断，不安装ROS或桌面客户端 |
| Git 远程仓库 | 源代码、配置、文档、数据集清单和模型校验值；不直接提交大型数据集和训练输出 |

这种“服务器集中运行 + 浏览器轻客户端”的方式只维护一套 ROS/GPU/Web 环境，避免 DDS、ROS、Python、CUDA 和 Node.js 在多台机器间不一致。系统部署完成后，普通操作员在 Windows 侧只打开 `http://ros-server/` 即可查看并控制整个巡检系统，不需要安装 ROS、Gazebo、Python、Node.js 或本项目代码。

### 3.2 可视化通道

1. **日常操作入口：** 自研Web综合控制平台，负责控制、感知、数字孪生、风险、历史和报告展示。
2. **三维展示：** Three.js在Windows浏览器本地利用WebGL渲染数字孪生，服务器只发送设备和机器人状态，不发送远程桌面画面。
3. **ROS诊断：** Foxglove Web仅供开发和排错，通过服务器上的 Foxglove Bridge 读取原始Topic、TF、地图和图像；它是独立的只读诊断旁路，不是普通操作员界面，不能发布命令或Topic，浏览器也不直接连接 DDS。
4. **Gazebo无头仿真：** Gazebo Harmonic通过OGRE2/EGL运行物理仿真和相机传感器，不启动Gazebo GUI，也不依赖X Server。
5. **命令行诊断：** 服务器Codex通过SSH、`ros2 topic`、`ros2 action`、`gz topic`、日志和rosbag2完成调试。

### 3.3 Codex 使用结论

项目只运行**一个 Codex 实例**：Ubuntu 服务器上的 Codex CLI。Windows 通过 SSH 进入服务器，并在 `tmux` 中操作这个 Codex 会话。因此从 Windows 发出的项目指令仍由服务器上的同一个 Codex 执行，使用同一个 Linux 文件系统、ROS 环境、上下文和运行状态。

Windows 本地安装的 Codex不参与本项目执行，也不需要和服务器 Codex通信。它可以保留用于其他 Windows 本地任务，但不得同时修改本项目仓库。该方案消除了两个独立 Codex 无法共享聊天上下文、可能重复执行命令或同时覆盖文件的问题。

## 5. 数据集方案

### 5.1 选型结论

没有一个可信的公开数据集能够同时覆盖变电站设备、人员、安全帽、烟雾、火焰、仪表以及多种设备异常。因此推荐采用“变电站设备主数据集 + PPE/烟火专项数据集 + 电力资产缺陷集 + Gazebo 仪表/异常合成数据”的组合。公开数据只作为用户外部训练参考，本仓库不下载；不同来源的标注语义和图像域差异较大，不强行训练成一个大检测模型，而是拆成安全检测、设备检测、缺陷分类和仪表读数四个可独立验证的模块。

### 5.3 模型拆分与类别映射

采用四个独立感知模块；仪表模块内部使用 YOLO11n 定位器和 OpenCV 后处理：

| 模块 | 模型与输出 | 数据来源 |
|---|---|---|
| 安全风险检测 | `yolo11n_safety.pt`：`person`、`no_hardhat`、`hardhat`、`fire`、`smoke` | Hard Hat Workers v10 + D-Fire |
| 变电站设备检测 | `yolo11n_equipment.pt`：15 类变电站设备 | 15-class Substation Equipment；InsPLAD-det 只补充语义可以明确对应的资产 |
| 设备缺陷分类 | `yolo11n-cls` 派生的 `yolo11n_fault.pt`：正常、腐蚀、部件破损、鸟巢等 | InsPLAD-fault + Gazebo 合成缺陷裁剪图 |
| 仪表读数 | YOLO11n 定位表盘，OpenCV 完成透视校正、圆盘/刻度/指针分析 | 仅使用 Gazebo 自动生成并标注的仪表数据 |

这样可以避免 15 类设备与大量烟火/PPE图像之间严重的类别不均衡，也避免把“目标检测框”和“设备裁剪图的缺陷分类”混进同一个输出头。温度、气体和设备上下文通过 ROS 传感话题进入风险融合模块，不伪装成视觉类别。

数据处理规则：

1. 保留各公开数据集原有训练/验证/测试边界，防止同源图片泄漏。
2. 将 COCO、VOC 或其他检测标注统一转换为 YOLO `class x_center y_center width height`；缺陷分类数据保持独立目录。
3. 所有类别建立唯一整数 ID，不以各来源原始 ID 直接拼接。
4. 合成数据按场景而不是按单张图片划分，测试场景的设备布局、光照和相机角度不出现在训练集。
5. 建立 `datasets/manifest.yaml`，记录来源链接、许可、revision、文件 SHA-256、类别映射和样本数。
6. 不直接重新发布来源数据；公开数据下载、转换和微调由用户在仓库外完成，仓库只保存来源参考、用户提供的 manifest、许可说明和校验结果。

### 5.4 YOLO11n 训练基线

```yaml
model: yolo11n.pt
imgsz: 640
epochs: 100
batch: 8
device: 0
workers: 6
seed: 42
patience: 20
```

该基线是用户外部微调产物的接纳要求。安全检测和设备检测分别训练；缺陷分类使用相同随机种子和独立的分类数据配置。RTX 3060 Ti 首先使用 `batch=8`；若显存不足则降为 4，不通过降低输入尺寸掩盖问题。用户发布包必须保存最佳权重、最终权重、混淆矩阵、检测模型每类 AP、缺陷模型 Balanced Accuracy、推理速度和数据版本清单，并通过不可变 GitHub release 或固定 commit 交付。

## 8. 关键 ROS 接口

优先复用标准消息：图像使用 `sensor_msgs/Image`，激光使用 `sensor_msgs/LaserScan`，目标框使用 `vision_msgs/Detection2DArray`，导航使用 Nav2 Action。仅为风险和巡检任务定义自有接口。

| Topic/接口 | 含义 |
|---|---|
| `/camera/image_raw` | Gazebo RGB 摄像头图像 |
| `/scan`、`/odom`、`/tf` | SLAM、定位和导航输入 |
| `/perception/detections` | YOLO11n 检测结果 |
| `/perception/annotated_image` | 带检测框和置信度的图像 |
| `/environment/temperature` | 带设备 ID 的模拟温度值 |
| `/environment/smoke` | 烟雾浓度或事件强度 |
| `/environment/gas` | 气体浓度或告警值 |
| `/digital_twin/assets` | 设备位置、类别、状态和最近证据 |
| `/risk/assets` | 各设备风险分数与风险等级 |
| `/risk/alerts` | 告警事件 |
| `/mission/inspection_tasks` | 当前巡检队列和优先级 |
| `NavigateToPose` | 向 Nav2 下发巡检目标 |

浏览器不直接连接 ROS DDS，也不直接发布 Topic；所有数据和命令必须经过 FastAPI Gateway。对外接口固定为：

| Web 接口 | 用途 |
|---|---|
| `GET /api/v1/system/status` | ROS节点、Gazebo、GPU、模型和服务健康状态 |
| `GET /api/v1/robot/state` | 位姿、速度、电量模拟值、当前模式和任务 |
| `GET /api/v1/assets` | 数字孪生设备、位置、状态和风险摘要 |
| `GET /api/v1/missions/current` | 当前巡检队列、目标和完成进度 |
| `POST /api/v1/missions/{action}` | `start`、`pause`、`resume`、`stop`、`return-home` |
| `POST /api/v1/navigation/goal` | 在地图点击后下发合法 Nav2 目标 |
| `POST /api/v1/robot/manual-velocity` | 仅在手动模式下发送限速速度指令 |
| `POST /api/v1/robot/emergency-stop` | 锁存紧急停止并取消当前 Nav2 任务 |
| `POST /api/v1/simulation/scenario` | 切换或触发指定 Gazebo 演示场景 |
| `GET /api/v1/reports` | 查询和下载巡检报告 |
| `WS /ws/telemetry` | 位姿、路径、传感器、风险和任务实时状态 |
| `WS /ws/events` | 告警、命令确认、任务变化和系统事件 |
| `WS /ws/camera` | 带检测框的 JPEG 二进制图像流 |

每个控制请求必须返回唯一 `command_id`、受理状态和时间戳；Gateway 在 ROS Action/Service 返回后再通过事件 WebSocket发送完成、失败或超时结果。

## 9. Gazebo 数字孪生设计

### 9.1 场景组成

- 一台主变压器；
- 断路器、隔离开关、避雷器；
- 电流互感器和电压互感器；
- 玻璃盘形和瓷针式绝缘子；
- 至少两个表盘仪表；
- 围栏、墙体、巡检通道和禁入区域；
- 静态障碍、移动人员和临时障碍；
- 可触发的烟雾源、温度热点和气体异常区域。

每个设备模型必须具有稳定的 `asset_id`、设备类别、地图坐标、正常阈值、巡检点位和报告名称。视觉模型负责“看见了什么”，Gazebo 场景事件负责提供可重复的仿真真值，两者不得在推理节点中直接互通，以避免模型读取答案。

### 9.2 场景事件

至少实现以下可配置测试场景：

1. 正常巡检；
2. 未佩戴安全帽人员进入；
3. 设备附近出现烟雾或火焰；
4. 变压器温度持续升高；
5. 气体浓度越限；
6. 仪表读数越限；
7. 高风险设备与动态障碍同时出现；
8. 目标巡检点不可达，任务管理器执行恢复或跳过。

## 10. 多模态风险融合

对设备 `i` 在时间 `t` 的风险定义为：

```text
R_i(t) = 100 × [0.30V + 0.25T + 0.20S + 0.10G + 0.15C]
```

其中：

- `V`：视觉风险，包括火焰、未佩戴安全帽、仪表越限和设备视觉异常；
- `T`：温度相对正常阈值的归一化超限程度；
- `S`：烟雾/火焰检测置信度与连续帧稳定度；
- `G`：气体浓度归一化值；
- `C`：设备重要度、人员接近程度及历史异常等上下文风险。

所有权重写入 `risk_weights.yaml`，不得硬编码。风险等级为：

| 分数 | 等级 | 行为 |
|---:|---|---|
| 0～29 | 正常 | 按普通路线巡检 |
| 30～59 | 关注 | 提高下一轮巡检频率并保存证据 |
| 60～79 | 告警 | 提升到当前任务队列前部并重新规划 |
| 80～100 | 紧急 | 中断普通任务，优先抵近安全观察点并发出告警 |

风险引擎采用连续帧确认、滑动平均和回落滞回，避免单帧误检导致机器人频繁改变任务。

## 11. 风险驱动巡检与路径规划

默认巡检路线由 `inspection_routes.yaml` 给出。任务优先级定义为：

```text
priority = base_priority + risk_gain × risk_score - distance_penalty × path_length
```

任务管理器每次出现以下事件时重新计算队列：

- 新设备风险达到 60 分；
- 当前设备风险等级发生变化；
- 动态障碍导致路径失败；
- 任务超时或目标不可达；
- 完成当前巡检点。

重新排序只改变巡检目标序列，机器人局部避障仍由 Nav2 完成。紧急设备的观察点必须位于预先定义的安全距离外，不能直接把目标设置在设备碰撞体内部。

## 12. Web 综合可视化控制平台

### 12.1 产品定位

Web平台是普通操作员的唯一日常入口，定位为“现代化变电站智能巡检控制中心”。它不是简单的数据看板，而是把机器人控制、实时感知、三维数字孪生、任务规划、风险告警、仿真场景和巡检报告集成到同一个浏览器应用中。

本期按单操作员、局域网部署设计。Windows端不安装任何项目依赖，只需浏览器；服务器承担页面渲染、ROS通信、数据库、图像转换和全部计算任务。

### 12.2 页面与功能

| 页面 | 核心内容 | 可执行操作 |
|---|---|---|
| 综合驾驶舱 `/` | 机器人状态、任务进度、最高风险、实时检测画面、缩略地图、关键告警 | 开始/暂停/继续/停止巡检、返航、紧急停止 |
| 三维数字孪生 `/twin` | 变电站三维模型、机器人实时位姿、已走路径、设备状态和风险颜色 | 点击设备查看证据，选择设备作为优先巡检目标 |
| 实时感知 `/perception` | 带检测框的视频、目标列表、温度、烟雾、气体、仪表读数和置信度 | 切换传感器/模型图层，冻结证据帧 |
| 地图与任务 `/mission` | SLAM占据地图、Nav2全局/局部路径、巡检点和任务队列 | 点击地图设目标、调整巡检点、重规划、切换手动/自动模式 |
| 风险中心 `/risk` | 设备风险排名、风险构成、多模态证据、历史曲线和告警时间线 | 确认告警、定位设备、发起复检 |

| 报告中心 `/reports` | 巡检记录、轨迹、告警、截图、模型和数据版本 | 在线预览，下载 HTML/PDF 和证据包 |

### 12.3 视觉与交互设计

- 采用深色工业控制台风格，主色为深蓝和青色，正常/关注/告警/紧急分别使用绿、黄、橙、红。
- 左侧为固定导航栏，顶部为服务器连接、机器人模式和任务状态，右侧为实时告警抽屉，底部固定任务控制栏和紧急停止。
- 综合驾驶舱默认同时显示实时视频、二维地图、机器人遥测、风险排行和任务进度，保证演示时无需频繁切页。
- 三维数字孪生使用 Three.js/React Three Fiber；设备按 `asset_id` 与 ROS语义状态绑定，风险变化时只更新材质和标记，不重新加载整个场景。
- 二维地图采用 Canvas/SVG 图层显示占据栅格、机器人、目标点、全局路径、局部路径和危险区域。
- 所有关键控制操作提供明确的等待、成功、失败和超时状态，不允许按钮点击后无反馈。
- 页面适配 1920×1080 演示屏，并兼容普通 1366×768 笔记本；手机端只保证只读概览，不提供机器人控制。

### 12.4 前后端数据流

```text
ROS 2 Topic / Service / Nav2 Action
  → substation_web_gateway（rclpy）
  → 状态聚合、单位转换、命令校验、图像JPEG编码
  → FastAPI REST + WebSocket
  → Nginx统一代理 /api 和 /ws
  → Next.js浏览器应用
```

Gateway按用途控制推送频率：机器人位姿和任务状态最高 10 Hz，风险与传感器 2～5 Hz，系统健康状态 1 Hz，带框视频目标为 10～15 FPS。前端只保存当前UI状态，长期历史写入服务器 SQLite；刷新页面后从 REST 快照恢复，再接续 WebSocket实时流。

### 12.5 控制确认与异常处理

- REST 控制请求先返回 `command_id` 和 `accepted`，最终完成状态通过 `/ws/events` 推送。
- Gateway连续发布心跳；浏览器超过 5 秒未收到心跳时进入“连接中断”状态并禁用普通控制。
- 手动速度控制只在手动模式有效，最大发送频率 10 Hz；松开按键、浏览器失焦或断线时立即发送零速度。
- 紧急停止为锁存状态：立即取消 Nav2任务、发布零速度并阻止新任务，必须执行单独的复位操作后才能继续。
- 地图点击目标必须经过可通行区域、设备碰撞体和危险区域检查，不合法目标在浏览器中直接标红并说明原因。
- 仿真场景控制只在 `simulation_mode=true` 时显示，避免未来接入真实机器人时误用。
- 视频、三维孪生或图表单独失败时不影响紧急停止和基础任务控制；页面显示降级状态并允许重新连接。

### 12.6 服务器部署与 Windows 使用

服务器使用 systemd 管理 `substation-web-gateway`、`substation-web-frontend` 和 Nginx。FastAPI只监听服务器回环地址，Nginx对局域网提供统一入口：

```text
http://ros-server/       Web综合控制平台
http://ros-server/api/   REST API（由前端调用）
ws://ros-server/ws/      WebSocket（由前端调用）
```

Windows日常使用流程只有三步：连接服务器所在局域网、打开现代浏览器、访问 `http://ros-server/`。SSH仅用于Codex开发和服务器管理；Foxglove Web仅通过服务器 Foxglove Bridge 提供只读 ROS 诊断，不能发布命令或Topic，也不直接连接DDS。二者都不是普通操作员完成巡检所必需的工具。服务器不提供远程桌面服务。

### 12.7 Web平台、Foxglove与无头Gazebo的关系

- Web平台：最终交付的产品化控制和展示界面；全部产品UI状态和控制都经 FastAPI ROS Web Gateway，承担三维数字孪生、二维地图、视频、风险和任务控制。
- Foxglove Web：通过服务器 Foxglove Bridge 在Windows浏览器中只读查看原始ROS Topic、TF、占据地图、路径和时间同步，定位数据链路问题；它不是普通操作员UI，不能发布命令或Topic，浏览器不直接连接DDS。
- Gazebo无头进程：负责物理、碰撞、传感器和场景事件，不提供GUI窗口。
- ROS/Gazebo CLI与rosbag2：验证Topic、Action、坐标变换、传感器频率、仿真时间和回放结果。
- 项目不部署RViz2、Gazebo GUI、Xorg或NoMachine；其诊断能力由Web平台、Foxglove Web和自动测试覆盖。

## 13. Codex 远程执行方式

### 13.1 唯一 Codex 实例

Codex CLI 安装并运行在 Ubuntu 服务器，项目仓库、ROS 2 工作空间、数据集、模型和运行产物也全部位于服务器。Windows只提供键盘、终端和图形显示，不在本地再启动第二个项目 Codex。

推荐操作链路：

```text
Windows Terminal
  → SSH 连接 Ubuntu 服务器
  → 进入 tmux 持久会话
  → 启动服务器 Codex CLI
  → Codex 直接操作服务器项目和运行环境
```

Windows 的 SSH 配置中为服务器设置固定别名 `ros-server` 后，使用以下命令连接：

```bash
ssh ros-server
tmux new -As codex
cd ~/substation_inspection
codex
```

SSH 暂时断开不会终止 `tmux` 中的 Codex、Gazebo 无头任务或批量验证进程。用户外部训练任务由用户自行选择持久运行方式，不属于本仓库执行职责。重新执行 `ssh ros-server` 和 `tmux new -As codex` 即可回到同一终端会话。

### 13.2 服务器 Codex 职责

服务器 Codex 对整个项目的实现和验证负责：

- 创建并维护 ROS 2 功能包、SDF、URDF/Xacro、launch、参数、测试和文档；
- 安装依赖，执行 `rosdep`、`colcon build`、单元测试和集成测试；
- 运行 Gazebo、Nav2、SLAM、YOLO11n、风险引擎和批量测试场景；
- 生成 Gazebo 仪表数据并提供可追溯 manifest；接收并验证用户训练产物，运行 GPU 推理并记录模型校验值；
- 检查 ROS 图、Topic、TF、日志、rosbag2、截图和性能指标；
- 启动Gazebo EGL无头仿真，验证相机Topic、仿真时间和物理状态，并通过Web页面或自动测试保存截图与录屏；
- 根据真实运行结果修复问题，提交 Git commit，并更新 `docs/PROJECT_STATUS.md` 和 `docs/HANDOFF.md`；
- 在服务器重启或上下文恢复后，根据仓库状态和交接记录继续任务。

只有服务器上的实际构建、测试和运行输出可以作为功能完成证据。

### 13.3 Windows 端职责

Windows端由用户操作，职责是：

- 日常运行时只通过浏览器访问 `http://ros-server/`，完成机器人控制、感知查看、风险处理和报告下载；
- 开发阶段通过 Windows Terminal 和 SSH 向服务器 Codex提出任务、查看进度并输入必要信息；
- 只有排错时才通过Foxglove Web检查原始ROS数据，或通过SSH查看服务器日志；
- 在需要时输入 `sudo` 密码、批准系统安装或驱动重启；
- 备份演示视频、报告和最终交付文件。

Windows 本地 Codex不是本项目的一部分，不与服务器 Codex同时操作仓库。如果以后确实需要独立审查，可以在服务器任务停止后让 Windows Codex读取某个 Git commit，但这属于离线复核，不是两个 Codex之间的实时通信。

### 13.4 权限和无头渲染边界

- Codex能够执行的服务器操作受 SSH 登录账户、`sudo` 权限、Codex 沙箱和网络设置约束。
- 系统软件、NVIDIA驱动和服务配置需要有权限的服务器账户；交互式密码或重启由用户配合。
- Gazebo通过OGRE2/EGL进行无头渲染，必须在没有`DISPLAY`的服务器环境中仍能发布相机图像和深度/激光数据。
- Codex通过Web截图、传感器图像、日志、Topic统计和rosbag2验证仿真，不依赖远程桌面窗口。
- 批量仿真和长时间验收应在 `tmux`、systemd 或持久任务脚本中运行，不能依赖普通 SSH 前台会话；模型训练由用户在仓库外自行管理。
- `AGENTS.md` 固定仓库结构、环境加载、构建测试命令、结果目录和完成标准；`PROJECT_STATUS.md` 与 `HANDOFF.md` 用于重启或上下文压缩后的恢复，而不是两端 Codex通信。

## 15. 测试与验收指标

### 15.1 环境与构建

- Ubuntu 24.04 上执行安装脚本后，`colcon build` 无错误。
- `nvidia-smi` 正常，PyTorch 返回 `torch.cuda.is_available() == True`。
- `npm ci` 和 `npm run build` 无错误，FastAPI Gateway测试通过，Nginx能够代理HTTP和WebSocket。
- `gz sim -s -r --headless-rendering substation_world.sdf`可在未设置`DISPLAY`的会话中运行，且相机Topic持续发布。
- 自研Web平台能够显示仿真数字孪生，Foxglove Web能够显示地图、TF、路径和ROS诊断数据。
- 服务器软件包清单和运行态证明不存在桌面元包、GNOME/KDE 会话、显示管理器、远程桌面、NoMachine、Xvfb、VirtualGL、活动 Xorg/Wayland 会话或项目配置的 X Server；允许 ADR-0004 规定的、由工作 NVIDIA 驱动需要的 inert X 包。

### 15.2 感知

- YOLO11n 在 640×640 输入下，服务器端完整 ROS 图像管线达到至少 15 FPS。
- 安全和设备检测模型在各自公开测试集上的总体 `mAP50` 目标不低于 0.75，同时必须报告各类别 AP；缺陷分类必须报告 Balanced Accuracy 和逐类召回率。
- 烟火和未佩戴安全帽告警采用连续帧确认，单帧误检不直接触发紧急任务。
- 每个检测结果可追溯到图像、时间戳、模型版本和数据集版本。

### 15.3 导航与风险闭环

- 正常场景连续 20 次巡检任务成功率不低于 90%。
- 风险达到 60 分后 2 秒内更新任务优先级，并向 Nav2 提交新的合法目标。
- 紧急场景中机器人不会把目标设置在设备或危险区域碰撞体内。
- 目标不可达时执行重试、替代观察点或跳过，并在报告中记录原因。
- 报告中包含巡检轨迹、设备状态、最高风险、告警证据、任务变更及模型版本。

### 15.4 Web平台

- 一台未安装 ROS、Gazebo、Python和Node.js的Windows电脑能够通过 `http://ros-server/` 使用全部日常功能。
- 综合驾驶舱、三维孪生、感知、任务、风险、场景、报告和系统状态八个页面均可正常打开。
- 位姿、任务和风险等遥测从服务器产生到浏览器更新的局域网P95延迟不超过500毫秒，带框视频端到端延迟目标不超过1秒。
- 控制请求在1秒内返回 `command_id` 和受理结果，完成、失败或超时最终状态可以追溯。
- 浏览器断开WebSocket超过5秒后禁用普通控制；恢复连接后通过REST快照和WebSocket继续更新。
- 手动控制松键、窗口失焦或断线时机器人停止；紧急停止能够取消Nav2任务并保持锁存。
- Playwright端到端测试覆盖启动巡检、暂停、继续、地图设点、场景触发、紧急停止和报告下载。

## 16. 资源预算

- RTX 3060 Ti 足以运行 YOLO11n 推理；用户外部训练基线的初始批量设为 8。
- Gazebo 和 Nav2 主要消耗 CPU，场景模型应控制网格复杂度和传感器更新频率。
- 名义 16 GiB 内存的主机在物理内存不少于 15 GiB 时可运行单机器人Gazebo无头仿真、YOLO11n和Web服务；项目批量推理与全速仿真分时运行以避免内存竞争。
- Next.js、FastAPI、Nginx和SQLite常驻服务器，前端生产构建不在Windows执行；模型集成或批量推理时可以暂时关闭三维页面预览，但Web控制和紧急停止保持可用。
- 服务器为项目预留不少于 80 GiB 可用空间，用于 Gazebo 合成数据、虚拟环境、用户模型资产、rosbag2 和报告；每次大资源阶段前必须独立记录资源投影，并在完成后至少保留 20 GiB，不能仅凭当前空闲空间声称足够。
- 日志和 rosbag2 按运行 ID 保存并设置保留策略，避免长期测试耗尽磁盘。

## 17. 风险与降级方案

| 风险 | 处理方式 |
|---|---|
| EGL无头渲染初始化失败 | 确认NVIDIA驱动、OGRE2渲染引擎和EGL可用；运行官方headless示例并检查渲染传感器日志，不以安装桌面环境规避问题 |
| 公开数据域与 Gazebo 图像差异大 | 项目生成带 manifest 的纹理、光照、视角、噪声和遮挡随机化样本，交给用户决定是否在仓库外再训练 |
| 类别不均衡 | 类别采样、合成数据和每类 AP 监控；不以总体 mAP 掩盖小类失败 |
| 多源数据许可不一致 | 在 manifest 中保存许可与引用；InsPLAD 仅限非商业使用；仪表数据完全由 Gazebo 生成，不引入外部仪表数据许可 |
| ROS Python 与 pip 包冲突 | `.venv --system-site-packages`、NumPy 精确锁定、禁止全局 `sudo pip` |
| WebSocket断线或消息积压 | Gateway按主题限频并只保留最新遥测；前端心跳超时后禁用控制并从REST快照恢复 |
| 浏览器视频占用过高 | JPEG流限制为10～15 FPS，按客户端状态降帧；关闭页面时停止订阅视频流 |
| Web控制误触或重复提交 | 危险操作二次确认、`command_id` 幂等、模式校验、超时处理和锁存急停 |
| 前后端接口版本不一致 | FastAPI生成OpenAPI，前端据此生成类型；CI同时验证Gateway契约和前端构建 |
| 风险值频繁跳变 | 连续帧确认、滑动平均、滞回阈值和最小任务保持时间 |
| 动态重规划造成路线振荡 | 设置重规划冷却时间，仅在风险等级变化或导航失败时重排 |
| 服务器资源不足 | 批量推理与仿真分时运行，降低非关键传感器和Web视频更新频率；用户外部训练资源由用户管理 |

7. 数字孪生、风险融合、告警和报告 ROS 2 节点；
8. FastAPI ROS Web Gateway、OpenAPI契约、SQLite结构和systemd服务；

## 19. 项目完成定义

只有同时满足以下条件才视为完成：

- 开发前文档门槛有明确检查记录，最终实现与项目计划、ADR、接口、版本和验收文档一致；
- 新服务器按照文档可重建环境；
- Gazebo、ROS 2、YOLO11n、风险融合和 Nav2 构成实际运行闭环；
- 高风险事件能够改变巡检优先级和机器人路径，而不只是界面上改变颜色；
- Windows只通过一个Web地址即可完成机器人监控、任务控制、风险查看、仿真场景和报告下载；
- Web三维孪生、二维地图、检测画面和风险状态与同一ROS运行时间线保持一致；
- 结果具有模型版本、数据版本、时间戳、rosbag2 和报告证据；
- 自研Web平台、Foxglove Web诊断和Gazebo无头仿真至少各完成一次验证；
- 所有交付物无未填写占位符，链接、命令、版本和文件路径经过复核。
