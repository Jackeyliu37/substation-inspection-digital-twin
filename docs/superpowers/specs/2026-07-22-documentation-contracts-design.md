# 阶段 0 文档与契约基线设计

**日期：** 2026-07-22  
**状态：** 已批准（用户授权采用推荐方案直接推进）  
**范围依据：** 根目录《基于数字孪生与多模态风险感知的变电站智能巡检系统——项目计划》

## 1. 目标

在任何功能代码、安装脚本或运行环境变更之前，建立一套能约束后续实现、支持零上下文恢复并可机械检查的文档基线。完成后，一名未参与前期讨论的开发者或 Codex 会话应能回答以下问题：

1. 项目做什么、不做什么；
2. 系统由哪些进程和 ROS 2 功能包组成；
3. 模块之间交换哪些消息、使用什么单位、QoS 和错误语义；
4. Web 如何读取状态、提交控制并收到最终确认；
5. 环境阶段应按什么顺序执行、如何判定失败和成功；
6. 当前仓库处于哪个阶段，恢复工作后的第一条命令是什么。

## 2. 方案比较与决定

### 方案 A：只建立文档目录和模板

优点是速度快，缺点是会留下大量待定字段，无法满足项目计划规定的开发启动门槛，也不能真正阻止实现阶段产生接口漂移。

### 方案 B：只优先冻结接口契约

优点是能尽早并行 ROS、Gateway 和前端工作，缺点是版本、部署、验收、数据治理和恢复入口仍不明确；单独的接口文档也缺少权威层级和变更规则。

### 方案 C：一次完成阶段 0 文档门槛（采用）

一次建立仓库规则、架构、部署、接口、版本、数据模型、验收、ADR、状态、交接和阶段 1 计划。成本高于前两种方案，但它与项目计划的阶段 0 定义完全一致，并把后续最昂贵的跨模块返工风险前移到文档阶段。

## 3. 文档体系与权威边界

文档分为五层，每份文档只有一个主要职责：

1. **范围层：** 根目录项目计划决定目标、范围、最终交付和总体验收。
2. **决策层：** `docs/adr/` 保存不可静默推翻的架构决定；变更通过新增 ADR 完成。
3. **规范层：** `ARCHITECTURE.md`、`INTERFACES.md`、`TEST_ACCEPTANCE.md`、`VERSION_MATRIX.md`、`DATA_AND_MODELS.md` 分别约束结构、契约、验证、版本和数据/模型治理。
4. **执行层：** `docs/plans/PHASE-01-ENVIRONMENT.md` 把环境阶段拆成可测试、可提交的任务。
5. **状态层：** `PROJECT_STATUS.md` 和 `HANDOFF.md` 只保存当前事实和恢复入口，不承载需求或架构决定。

`AGENTS.md` 是自动加载的导航与执行规则，不复制长篇设计。`README.md` 是人员入口，提供项目概览和文档导航。

当文档冲突时，优先级为：用户当前明确指令、项目计划、ADR、专项规范、阶段计划、状态与交接文档。发现冲突时先修正文档，暂停受影响实现。

## 4. 架构边界

系统保持以下单向主数据流：

```text
Gazebo/机器人传感器
  -> 感知与传感归一化
  -> 语义数字孪生
  -> 风险引擎
  -> 任务管理器/Nav2
  -> FastAPI ROS Web Gateway
  -> Next.js 浏览器应用
```

关键隔离规则：

- Gazebo 场景真值只进入仿真测试与传感器模拟，不直接进入视觉推理结果；
- 风险引擎只消费带时间戳和设备标识的归一化证据，不依赖 Web 状态；
- 任务管理器只通过 Nav2 标准 Action 下发目标，不自行实现局部规划；
- 浏览器不连接 DDS，不发布 ROS Topic；所有控制经过 Gateway 校验和审计；
- Gateway 负责协议转换、限频、快照与异步命令确认，不承担风险计算或任务排序；
- SQLite 保存历史和索引，实时 ROS 状态仍以 ROS 图为权威来源。

## 5. 契约设计

### 5.1 ROS 2 契约

优先使用标准类型：图像 `sensor_msgs/msg/Image`、激光 `sensor_msgs/msg/LaserScan`、检测框 `vision_msgs/msg/Detection2DArray`、导航 `nav2_msgs/action/NavigateToPose`。自定义接口仅覆盖设备证据、数字孪生状态、风险、告警、巡检任务和任务控制。

所有自定义状态消息必须包含 ROS 时间戳；设备相关消息必须包含稳定的 `asset_id`。物理量使用 SI 单位：位置为米、角度为弧度、速度为米每秒/弧度每秒、温度为摄氏度、气体为 ppm、烟雾为 0～1 无量纲强度、风险为 0～100。

状态 Topic 使用命名空间 `/substation` 之外的项目计划既定名称，避免后续重复迁移。传感器和高频状态采用 best-effort/volatile；任务、风险和告警采用 reliable；需要新订阅者立即获得最后快照的聚合状态采用 transient-local。

### 5.2 Web 契约

REST 使用 `/api/v1`，返回 JSON。所有控制请求立即返回 HTTP `202`、唯一 `command_id`、`accepted_at` 和当前受理状态；最终的 `succeeded`、`failed`、`timed_out` 或 `cancelled` 通过 `/ws/events` 发布。重复提交使用 `Idempotency-Key`，Gateway 在 24 小时内返回相同命令记录。

WebSocket 消息统一使用信封字段 `schema_version`、`type`、`sequence`、`timestamp`、`run_id` 和 `payload`。客户端发现序号缺口或重连时必须重新获取 REST 快照。心跳间隔为 1 秒，超过 5 秒未收到心跳则禁用普通控制。

错误响应使用 RFC 7807 风格字段，并增加稳定的项目错误码。急停是锁存命令，复位使用独立端点；任何普通任务命令都不能隐式解除急停。

## 6. 数据流和状态所有权

| 状态 | 唯一写入者 | 主要消费者 | 持久化 |
|---|---|---|---|
| 仿真设备真值 | Gazebo 场景系统 | 传感器模拟、场景测试 | 场景配置与 rosbag2 |
| 视觉检测 | 感知节点 | 数字孪生、证据记录、Gateway | 证据索引与图像 |
| 设备当前状态 | 数字孪生节点 | 风险、Gateway、报告 | SQLite 时间序列 |
| 风险分数与等级 | 风险节点 | 任务管理器、Gateway、报告 | SQLite 事件与时间序列 |
| 当前任务队列 | 任务管理器 | Gateway、报告 | SQLite 任务记录 |
| Nav2 执行状态 | Nav2/任务管理器 | Gateway、报告 | 任务事件与 rosbag2 |
| 命令生命周期 | Gateway | 浏览器、审计、报告 | SQLite 命令表 |

设备状态更新按 `asset_id` 合并。消息时间戳早于当前设备字段最后更新时间的更新不得覆盖新值，但可保留为历史证据。跨进程事件都携带 `run_id`，使报告、rosbag2、截图和数据库记录可关联到同一次巡检。

## 7. 故障与降级

- 摄像头或模型不可用时，视觉风险标记为不可用而不是零；温度、烟雾和气体链路继续运行。
- Gateway 与浏览器断线不停止 ROS 自主任务，但浏览器进入只读降级并禁用普通控制；急停端点保持最高优先级。
- SQLite 写入失败时不阻塞急停和 Nav2 控制，但发布持久化降级告警并禁止生成“完整”报告。
- Nav2 目标失败时任务管理器按重试、替代观察点、跳过的顺序处理，并记录原因。
- 风险数据陈旧时按配置的最大年龄标记 `stale`；任务管理器不得把陈旧分数当作新告警触发重排。

## 8. 验证策略

阶段 0 采用静态门槛验证：检查必需文件、权威链接、禁止占位符、契约关键字、版本行、ADR 状态和阶段 1 入口。后续代码阶段遵循测试先行：单元测试验证纯逻辑，ROS launch 测试验证节点与 QoS，场景测试验证闭环，Playwright 验证浏览器控制和故障状态。

证据统一写入 Git 忽略的 `artifacts/<run_id>/`，至少包含命令日志、JUnit/XML、Topic/TF 快照、性能结果、截图、rosbag2 和验收摘要。完成声明必须引用实际命令与证据路径。
